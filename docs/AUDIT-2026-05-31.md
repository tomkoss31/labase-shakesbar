→ **Reco** : créer `api/_lib/loyalty.ts` (constantes + fonctions) et l'importer dans `orders.ts` et `profile.ts`. Laisser `square-webhook.ts` tel quel pour l'instant (fichier fragile) ou le migrer en toute fin, testé.

### Catalogue récompenses dupliqué

## 🔴 Sensible — Square (NE PAS toucher sans session dédiée + test e2e)

Ces points sont **réels** mais touchent le flux de paiement fragile. À traiter prudemment, un par un, avec un test de paiement complet derrière.

1. **Ordre débit XP / création du lien Square** (`api/create-payment-link.ts`) : aujourd'hui on **débite les XP AVANT** d'appeler Square. Si Square échoue après, les XP sont perdus (pas de rollback).
   → Reco : créer le lien Square D'ABORD, puis débiter les XP seulement si le lien est OK.
2. **Code promo roue marqué "utilisé" APRÈS** la création du lien (best-effort) : sur un retry, un même code pourrait passer 2×.
   → Reco : marquer `used_at` AVANT de créer le lien.
3. **Pas de table d'audit XP** (`xp_transactions`) : aucun journal des débits/crédits → difficile de réconcilier en cas de litige.
4. **Idempotence** : pas de contrainte SQL unique sur (user_id, square_order_id) côté `orders` → théoriquement double-crédit possible si webhook + mark-paid collisionnent.
5. **Bonus combo +25** : présent au crédit manuel comptoir, absent du webhook carte et de mark-paid (le total seul ne dit pas si c'est un combo). Décision métier à trancher.
